# 元技能评分卡（创作者版 v3.0）

使用条件：安全门禁 ✅ 通过 + 类型确认为元技能。

元技能是用来创建/评估/优化其他技能的技能。它的用户是技能创作者，而非一般消费者。

---

## 维度 1：覆盖度（25 分）

### 自动检查
- 阅读 SKILL.md 查看是否覆盖以下链路：**构思 → 写草稿 → 写测试 → 运行 → 评估 → 迭代 → 打包发布**
- 检查是否有"测试"、"evals"、"benchmark"、"迭代"、"打包"等覆盖关键词

### 打分

| 分 | 覆盖链路数 | 说明 |
|:--:|:---------:|:-----|
| 25 | 7/7 | 完整：构思→测试→评估→迭代→打包发布 |
| 21 | 6/7 | 缺末尾（打包发布）或开头（构思引导） |
| 17 | 5/7 | 缺测试或迭代环节 |
| 12 | 3-4/7 | 只有创建+简单测试 |
| 6 | 2/7 | 只能创建草稿 |
| 0 | 0-1/7 | 说自己是元技能但做不了什么 |

---

## 维度 2：可测试性（20 分）

### 自动检查
- 搜索 `eval`、`benchmark`、`test`、`assert`、`grader` 等关键词
- 检查是否有评估脚本（`run_eval`、`aggregate_benchmark` 等）
- 检查是否有 `evals/` 目录或 fixtures

### 打分

| 分 | 条件 |
|:--:|:------|
| 20 | 完整 evals 框架：测试集 + assertions + benchmark + 可视化比较 |
| 17 | 有评估框架脚本 + 部分测试集 |
| 13 | 有框架脚本但无测试集 |
| 10 | 只有手动测试指导 |
| 5 | "你自己看效果" |
| 0 | 不提测试 |

---

## 维度 3：结构化（15 分）

### 自动检查
- 检查产出物的结构规范：是否要求 SKILL.md ≤ 500 行 + references/ + agents/
- 检查元技能自己的 SKILL.md 行数（以身作则）

### 打分

| 分 | 对产出物的要求 |
|:--:|:--------------|
| 15 | 严格渐进披露规范 |
| 12 | 有结构意识但执行不严 |
| 8 | 知道要分文件但没标准 |
| 4 | 产出单一大文件 |
| 0 | 完全无结构要求 |

**自己行数调整**（以身作则原则）：
- 自己 ≤ 350 行：+3（上限 18）
- 自己 351-500 行：不加不扣
- 自己 501-800 行：-3
- 自己 800+ 行：-5
- 自己 1200+ 行：-8

## 维度 6：流程完整性（10 分）

**v3.0：** 不再测量"管得多严"，而是测量"流程多完整"——Claude 跟着走就能出正确结果。

### 自动检查
- 搜索是否有阶段/步骤编号
- 搜索是否覆盖完整链路（输入→处理→决策→输出→验证）
- 搜索条件分支、错误处理、退路

### 打分

| 分 | 条件 |
|:--:|:------|
| 10 | 完整可执行流程：分阶段 + 决策点 + 分支处理 + 退路 |
| 7 | 有流程但缺少分支处理或退路 |
| 4 | 有流程框架但多处需要 Claude 自己补 |
| 0 | "随你便" |
